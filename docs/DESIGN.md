# Oasis — 游戏设计文档
> 版本 0.9 · 设计定稿，待开发

---

## 核心定位

**一款 AI 驱动的文字 MUD，玩家可以创造任何世界，也可以一句话立即开玩。**

竞品现状：AI Dungeon / Friends & Fables / NovelAI 均为英文市场，固定世界观或固定规则系统。  
Oasis 的差异化：**规则本身由玩家世界观推导，中文优先，创造与游玩分离。**

---

## 核心设计原则

1. **AI 永远填补空白** — 玩家没定义的部分，AI 推断并填入，绝不报错或留白
2. **任意层可开始游戏** — 创造深度不影响入场，10秒也能开玩
3. **World Bible 是契约** — 所有 AI 行为受 World Bible 约束，保证世界一致性
4. **创造者与冒险者可以是不同人** — 设计世界和玩世界是两种乐趣
5. **玩家历史是公开的** — 所有玩家的行动留痕于世界，后来者能感受到前人的足迹
6. **历史是世界的一部分** — 没有历史的世界是空洞的布景，World Bible 必须包含活的历史

## 世界创造：洋葱模型

同一套系统，三层入口，玩家在任意层停下即可开始游戏。

```
第 0 层  一句话（10秒）
─────────────────────────────────────────
输入：  "赛博朋克东京，武士与黑客并存"
输出：  AI 自动生成完整 World Bible 默认值
立即开玩

          ↓ 可选：深入定制

第 1 层  关键选择（3分钟）
─────────────────────────────────────────
几个核心维度，每项 AI 提供 3 个建议选项
可直接采纳，也可自由输入
· 世界基调
· 主要冲突
· 玩家角色定位
· 死亡规则

          ↓ 可选：深入定制

第 2 层  完整 World Bible 编辑（无上限）
─────────────────────────────────────────
给"创造者型玩家"的完整控制权
可编辑所有字段，AI 实时预览世界效果
```

## 世界锻造台 UX ✅ 已决策

### 第 0 层：一句话输入后
AI 生成 World Bible，呈现**开场白 + 世界摘要**，同时提示"想深入定制吗？"引导进第 1 层。不直接跳游戏，给玩家感受世界氛围的机会。

### 第 1 层：关键选择
**AI 对话流形式**——AI 逐步提问，一次一个维度。
- AI 在回答中自然带出进度感（"还有两个问题…" / "最后一个——"）
- AI 主动告知玩家可以随时输入"跳过"或"直接开始"
- 不用表单、不用进度条，沉浸感优先

```
AI：你的世界笼罩在怎样的基调下？
    是绝望中的余烬，还是帝国最后的黄昏，
    或者完全是你自己的感觉？
    （还有 3 个问题，随时可以说"跳过"直接开始）

玩家：末日感，但有一丝荒诞的幽默

AI：有意思的组合。那么——这个世界里，
    死亡是终点吗？还是另一种开始？
    （还有 2 个问题…）
```

### 第 2 层：完整编辑器
- 对玩家友好的结构化编辑界面（非裸 JSON）
- **实时预览**：AI 根据当前 World Bible 状态生成一段示例叙事，让创造者即时感受世界氛围
- 每次修改关键字段后自动刷新预览

## 角色创建

角色完全由玩家自定义，同样遵循洋葱模型——可以一句话定义，也可以深度设定。  
角色定义存入玩家档案，跨 session 持久化，与世界历史产生交集。

```json
"player_character": {
  "name": "自定义",
  "concept": "一句话：落魄的前帝国侦探",
  "background": "可选的详细背景",
  "traits": ["执着", "酗酒", "曾经是英雄"],
  "starting_resources": "由世界规则推导或玩家指定",
  "secret": "只有玩家知道、AI 主持人据此叙事的隐藏设定"
}
```

`secret` 字段是一个有趣的设计——玩家可以设定一个世界内其他人不知道的秘密（"我其实是反派的儿子"），AI 主持人会在叙事中埋下对应的伏笔，但不会直接说破。

## 历史系统

> 历史不是附加功能，它是世界本身。World Bible 必须包含活的、会生长的历史。

### 三种历史，三种处理方式

| 类型 | 范围 | 存储方式 | 处理策略 |
|------|------|----------|----------|
| 对话历史 | 当前 session | 内存 | 滚动压缩，旧对话→摘要 |
| 玩家档案历史 | 跨 session | 数据库结构化 | 提取关键状态，每次注入 |
| 世界历史 | 全局公开 | World Bible 历史字段 | 分层压缩 + 向量检索 |

### 分层历史（核心设计）

越久远的历史分辨率越低——这符合真实世界逻辑，也解决了上下文窗口限制问题。

```
 久远 ←─────────────────────────────→ 最近

 [神话层]        [编年史层]        [当代层]
 极度压缩，        中度压缩，        完整保留，
 数百字            数千字            全量事件

 "传说中          "三年前           "昨天
  第一个玩家        王都被焚毁，       player_07
  推翻了旧帝国"     北方派系崛起"      发现了地下室"
```

**压缩机制**：定时任务定期把当代层事件压缩进编年史层，再把编年史层压缩进神话层。细节逐渐抽象，但重要事件永不丢失。

### 世界历史数据结构

```json
"world_history": {
  "myth_layer": "极度压缩的创世传说与远古大事，数百字以内",

  "chronicle_layer": [
    {
      "era": "第一纪元",
      "summary": "时代压缩摘要",
      "landmark_events": ["王国建立", "大战爆发", "英雄陨落"]
    }
  ],

  "recent_layer": [
    {
      "event_id": "evt_0042",
      "world_time": "第7天第3小时",
      "player": "ghost_01",
      "action": "摧毁了王都核心反应堆",
      "consequence": "北方三城断电，流亡者涌入南境",
      "world_bible_patch": {
        "society.factions.北方联盟.power": -30,
        "generated_lore.key_locations.王都.status": "废墟"
      }
    }
  ]
},

"world_state_snapshot": {
  "version": 42,
  "last_updated": "timestamp",
  "active_players_count": 3,
  "summary": "AI 生成的当前世界状态一句话描述"
}
```

`world_bible_patch` 是关键设计：每个历史事件直接描述它对 World Bible 其他字段的修改。AI 不需要重读所有历史，只需读当前快照就能知道世界现在的样子。

### AI 调用时的历史注入策略

不是每次都塞入完整历史，而是**按场景相关性动态选取**：

```
每次 AI 调用 prompt 组装：

  1. World Bible 核心规则        （固定，精简版）
  2. 世界状态快照                （当前快照，不含历史过程）
  3. 近期重要事件摘要             （最近 N 条）
  4. 场景相关历史片段             （向量检索：当前地点/派系相关）
  5. 玩家角色档案                （含跨 session 状态）
  6. 当前 session 对话            （滚动窗口，旧→摘要）
```

玩家在王都时只加载王都相关历史，不加载海边城市的事——**按需检索而非全量注入**。

## 多人架构 ✅ 已决策

### 路线图

**Phase 1（MVP）：小团队实时共存（2-8人）**
**Phase 2（扩展）：异步大世界（不限人数）**

两阶段共享同一套底层基础设施，Phase 2 是 Phase 1 的自然延伸。

---

### Phase 1：小团队实时架构

#### 世界时间模型

```
IDLE   无玩家在线，世界时间冻结
       world_bible_patch 暂存队列，不执行

ACTIVE 有玩家进入，世界恢复运转
       暂存 patch 按顺序结算，历史正常写入
```

优点：无后台 AI 空转，成本可控。

#### 主持人 + Agent 弹性扩容模型

主持人是**唯一的世界状态权威**，Agent 只负责叙事生成，没有独立判断权。

```
正常负载（队列短）
  玩家 A ──┐
  玩家 B ──┤──→ 主持人（完整上下文 + 世界裁决）
  玩家 C ──┘

队列积压超过阈值（如等待 > 3秒）
  玩家 A ──→ Agent #1 ──→ 叙事草稿 ──┐
  玩家 B ──→ Agent #2 ──→ 叙事草稿 ──┤──→ 主持人裁决 ──→ 返回玩家
  玩家 C ──→ Agent #2 ──→ 叙事草稿 ──┘
```

**职责分离**

| 角色 | 职责 | 特点 |
|------|------|------|
| 主持人（Master） | 持有 world_state、裁决所有 patch、广播世界事件 | 串行、权威、轻量 |
| Agent（Workers） | 叙事生成、上报意图 | 并行、可扩容、无状态 |

**Agent 上报意图，而非直接写状态**

```json
{
  "player": "ghost_01",
  "intent": "摧毁东城门",
  "proposed_patch": { "王都.东城门.status": "destroyed" },
  "narrative_draft": "你引爆了炸药，城门轰然倒塌..."
}
```

主持人收到后：验证合法性 → 检查冲突 → 裁决写入 → 返回确认叙事。

**一致性由主持人保证，Agent 不参与裁决。**

**扩容规则**

```
Agent 数量 = ceil(在线玩家数 / 3)
队列等待 > 3秒  → 触发扩容
空闲 > 30秒    → 回收多余 Agent
```

**Agent 注入的上下文（精简版，非完整对话历史）**

```
world_state 快照        当前版本
近期世界事件            最近 N 条 patch
各玩家当前状态          位置 / HP / 关键 flags
玩家间关系摘要          "A 和 B 合作，C 是对立方"
```

---

### Phase 2：异步大世界架构（后期扩展）

核心机制：**版本号 + 事件队列 + AI 仲裁**

```
玩家行动 → 提交 { action, base_version } → 队列处理器
  ├── base_version 匹配 → 直接应用，版本 +1
  ├── 无冲突 → 合并应用
  └── 有冲突 → 主持人 AI 仲裁，生成合并 patch

玩家体验层：
  前人的 patch 变成世界的背景叙事
  "你回到王都。空气中还有硝烟——
   三天前有人炸毁了东城门，那个人叫 ghost_01。"
```

Phase 1 → Phase 2 的主要新增组件：`事件队列（Redis Streams）+ 版本号机制 + 冲突仲裁逻辑`，其余复用。

### 玩家互动 ✅ 已决策

**可见性**
- Phase 1（小团队）：队友行动完全实时可见
- Phase 2（大世界）：按空间范围可见，范围外事件通过**新闻系统**感知

**新闻系统（Phase 2）**
范围外的世界事件不直接推送，而是经过世界过滤后以叙事形式呈现：NPC 口耳相传、酒馆传言、告示板通缉令、派系公告……信息可以失真、延迟、带有派系立场。与分层历史系统天然对接——当代层事件 → 新闻扩散 → 逐渐沉淀进编年史层。

**合作**
- 支持组队 + 一致行动（AI 主持人同步处理队伍行动）
- 多团队之间的交互后期再设计

**PvP**
- Phase 1 不做
- Phase 2 由 World Bible 定义：世界创造者决定是否允许、规则如何，AI 主持人执行

## 设计备忘与风险点

### 需要提前解决的问题

**World Bible 约束力漂移**
AI 在长对话后会偏离 World Bible 规则（比如世界规定"死亡永久"，打了 50 回合后 AI 悄悄给玩家留后路）。建议在 prompt 结构里加入**约束自检步骤**——AI 生成回应前先检查："我即将做的事违反 World Bible 的哪条规则吗？"不需要额外 API 调用，是 prompt 内的思维步骤。MVP 阶段先用简单世界测试约束力边界，再逐步开放创造复杂度。

**第 1 层跳过后的状态**
玩家跳过某个维度时，该维度用 AI 默认值填充。这套填充逻辑要提前明确，否则 AI 会在游戏里自己发明规则，玩家误以为那是世界设定。

### 被低估的设计点

**开场白质量决定留存率**
第 0 层之后的 AI 开场白是玩家对游戏的第一印象。需要专门设计 prompt，目标不是"描述世界"，而是"用最能让人心跳加速的方式，把玩家扔进世界的第一幕"。

**角色 `secret` 字段的叙事潜力**
玩家写下个人秘密（"我是这个世界 AI 觉醒的第一个受害者"），AI 主持人据此在叙事里埋伏笔，其他玩家无感知，只有这个玩家能感受到世界在回应他的身份。这是文字游戏里很难做到的个人叙事感，MVP 里值得认真打磨。

## 世界创造者定位 ✅ 已决策

**创造者也是玩家，以普通玩家身份进入自己创造的世界。**

**设定公开性**
- 创造者的所有设定（World Bible 全部内容）**对所有玩家公开**
- 没有上帝视角，没有信息优势

**隐藏设定**
- 由 AI 在世界创造时**随机生成**，创造者本人也不知道
- 是 AI 埋入世界骨架里的秘密，等待任何玩家去发现
- 创造者第一次进入自己的世界，同样是在探索和发现

这个设计解决了两个问题：
1. **平衡性**：创造者不因"知道答案"而破坏体验
2. **创造者动机**：设计世界本身有乐趣，进入后仍有惊喜

**隐藏设定生成时机 ✅**：创造时一次性生成并锁死。世界从诞生起秘密就存在，只是没人知道——像地底的矿脉，不挖就不知道，但它一直在那里。

- 世界公开后，其他玩家进入时创造者获得积分/声望
- 玩家可以 fork 别人的世界，基于它再创造
- 热门世界排行榜

## MVP 范围 ✅ 已定稿

核心流程四步，范围之外全部后期。

```
创造世界 → 创建角色 → 进入游戏 → 状态持久化
```

### 第一步：创造世界
- 第 0 层：一句话输入 → AI 生成 World Bible → 开场白 + 摘要 → 引导进第 1 层
- 第 1 层：AI 对话流定制（世界基调/主要冲突/死亡规则等），可随时跳过
- 第 2 层：砍掉，后期实现
- AI 同时生成隐藏设定，创造时锁死，所有人（含创造者）不可见

### 第二步：创建角色
- 对话式，AI 引导
- 基础字段：名字、概念、背景、特质
- `secret` 字段：玩家填写，只有 AI 主持人知道，其他玩家不可见

### 第三步：游戏终端
- 已有 CRT 原型，接入 World Bible 约束
- AI 主持人严格遵守 World Bible 规则（含约束自检 prompt 步骤）
- 小团队实时，队友行动完全可见
- 支持组队 + 一致行动

### 第四步：跨 session 持久化
- 玩家状态存库（HP / 位置 / 物品 / 剧情 flags）
- World Bible 存库（含隐藏设定）
- 角色档案存库
- 重启可续玩
